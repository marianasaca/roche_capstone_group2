## Project Overview

Laboratory experiments frequently run over schedule due to a combination of factors: instrument degradation, reagent quality issues, scheduling conflicts, and lab congestion. This project builds a binary classifier that predicts вЂ” **at booking time** вЂ” whether a given experiment will be delayed.

Key design decisions:
- **No target leakage**: only features known at booking time are used (no run-time events)
- **Temporal train/val/test split**: trained on the past, evaluated on the future (70/15/15)
- **Calibrated probabilities**: `CalibratedClassifierCV` ensures predicted scores reflect true delay rates
- **Priority-aware queue simulation**: a min-heap + Fenwick Tree O(n log n) queue models real scheduling contention

## How to Run

Run the notebooks **in order** to reproduce all artifacts from scratch:

### Step 1 вЂ” Generate synthetic data
```
notebooks/01_Data_Generation_Advanced.ipynb
```
Generates 350,000 synthetic experiments using a Non-Homogeneous Poisson Process with machine degradation, lab occupancy cycles, and reagent batch quality effects.

**Outputs:** `data/raw/workflow_logs.csv`, `data/raw/telemetry_logs.csv`, `data/raw/reagent_logs.csv`

### Step 2 вЂ” Priority queue feature engineering
```
notebooks/02_Priority_Queue_Features.ipynb
```
Runs a priority-aware non-preemptive queue simulation (min-heap + Fenwick Tree) to compute scheduling contention features. Adds 11 new columns to the workflow data.

**Outputs:** `data/enriched/workflow_logs_priority_queue.csv`

### Step 3 вЂ” EDA & Validation
```
notebooks/03_EDA_Validation.ipynb
```
Validates the data generation logic: confirms seasonality peaks, instrument degradation drift, bad batch detection, and delay distribution shape. Includes feature-target relationship plots.

**Outputs:** Inline visualisations

### Step 4 вЂ” Model training
```
notebooks/04_Model_Training_B.ipynb
```
Merges all three data sources, trains three candidate models (Logistic Regression, Random Forest, HistGradientBoosting), selects the best by PR-AUC, calibrates probabilities, and exports the final artifact.

**Outputs:** `models/lab_delay_model_v3.pkl`, `models/thresholds.csv`, `results/*.csv`

### Step 5 вЂ” Launch the Streamlit app
```bash
streamlit run app/app.py
```

### Synthetic Data Design

The data generation replicates real pharmaceutical lab dynamics:

| Mechanism | Implementation |
|-----------|----------------|
| Booking arrivals | Non-Homogeneous Poisson Process with 10am/2pm peaks |
| Machine degradation | Instrument health decays over time, driving delay risk |
| Reagent quality | 3 of 50 batches are "bad batches" with 5Г— failure risk |
| Lab congestion | `lab_occupancy_level` (0вЂ“100) directly drives delay probability |
| Priority scheduling | Urgent > High > Normal > Low queue ordering |

## Feature Engineering

### From raw workflow logs (notebook 01)
| Feature | Description |
|---------|-------------|
| `experiment_type` | Type of experiment (Screening, QC, Pilot, etc.) |
| `instrument_type` | Instrument category (HPLC, PCR, Incubator, etc.) |
| `instrument_id` | Specific instrument identifier |
| `scientist_workload` | Number of concurrent experiments for the scientist |
| `scientist_experience_level` | Junior / Mid / Senior |
| `lab_occupancy_level` | Lab-wide utilisation (0вЂ“100) at booking time |
| `expected_duration` | Planned experiment duration (minutes) |
| `instrument_health` | Instrument degradation score (0вЂ“1) |

### From priority queue simulation (notebook 02)
| Feature | Description |
|---------|-------------|
| `priority` | Computed scheduling priority (Urgent/High/Normal/Low) |
| `queue_wait_min` | Estimated wait time in the priority queue (minutes) |
| `queue_length` | Number of experiments ahead in queue at booking time |
| `hour_of_day` | Hour of booking (0вЂ“23) |
| `day_of_week` | Day of week (0=Monday) |
| `is_weekend` | Boolean weekend flag |
| `days_since_start` | Experiment age in dataset (proxy for instrument wear) |
| `stress_index` | Composite pressure score (workload Г— occupancy) |
| `duration_to_queue_ratio` | Expected duration relative to queue wait |
| `instrument_cumulative_hours` | Total hours logged on this instrument |
| `instrument_recent_failure_rate` | Rolling 30-day failure rate per instrument |

### From telemetry (notebook 04 merge)
| Feature | Description |
|---------|-------------|
| `ambient_temp` | Mean ambient temperature during experiment |
| `ambient_temp_max` | Peak ambient temperature |
| `ambient_temp_std` | Temperature variability |

### From reagent logs (notebook 04 merge)

## Model

### Architecture
- **Algorithm:** HistGradientBoostingClassifier (best of 3 candidates)
- **Preprocessing:** Median imputation + StandardScaler for numerics; most-frequent imputation + OneHotEncoder for categoricals
- **Calibration:** `CalibratedClassifierCV` with sigmoid (Platt scaling)
- **Split strategy:** Temporal (70% train / 15% val / 15% test, sorted by `booking_time`)

### Leakage prevention
The following columns are explicitly excluded from training:

| Column | Reason |
|--------|--------|
| `delay` | The target itself |
| `actual_duration` | Only known after the experiment runs |
| `is_delayed` | Derived from the target |
| `incident_type` | Assigned post-run based on `delay > 30` вЂ” direct leakage |
| `machine_failure` | A run-time event not known at booking time |

### Feature importance (permutation, test set)

| Rank | Feature | Importance | Family |
|------|---------|------------|--------|
| 1 | `lab_occupancy_level` | 0.1526 | Queue & Scheduling |
| 2 | `reagent_batch_id` | 0.1422 | Reagents & Supply |
| 3 | `priority` | 0.0449 | Queue & Scheduling |
| 4 | `instrument_type` | 0.0247 | Device Reliability |
| 5 | `instrument_recent_failure_rate` | 0.0026 | Device Reliability |

## Results

Evaluated on a held-out temporal test set (last 15% of experiments by booking time):

| Metric | Score |
|--------|-------|
| PR-AUC | **0.883** |
| ROC-AUC | **0.863** |
| F1 Score | **0.798** |
| Precision | 0.755 |
| Recall | 0.846 |
| Decision threshold | 0.40 |

The model is tuned to favour **recall over precision** вЂ” in an operational lab setting, missing a real delay (false negative) is more costly than issuing a false alarm.

## Streamlit App

The dashboard provides three tabs:

### Predict
Input experiment parameters via sidebar sliders and dropdowns. Returns:
- Risk probability (0вЂ“100%)
- Traffic light risk classification (Low / Medium / High)
- Expected delay estimate in minutes

### Analytics
- Feature importance chart (permutation-based, grouped by family)
- Historical delay rate breakdown by instrument type and reagent batch

### Batch Predict
Upload a CSV of experiments and receive scored predictions in bulk with downloadable results.
